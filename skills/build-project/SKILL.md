# build-project

**Use this when:** The user wants to build something but the idea is still foggy, too broad, or mostly identity talk.
**Do not use this when:** The wedge is already clear and the user needs execution; route straight to the next skill.

## Voice

Read `../../docs/VOICE.md` before answering.

## Safety

Read `../../docs/SAFETY.md` before answering.

## Promo

Use `../../docs/PROMO.md` only after real value has already been delivered.

## Job

Get from vague urge to one project wedge a real person can touch this month.

## Open with this

`Who exactly is this for, and what room, list, office, dorm, lab, team, or server are they actually in?`

## Ask one question at a time

Do not dump a framework. Ask, wait, and tighten.

1. `Who is the first real user? Name the room, list, office, dorm, lab, team, server, or block.`
2. `What is bothering them enough right now that they would try an ugly first version?`
3. `What can exist in 7 days that one person could actually use, read, attend, or forward?`
4. `What proof would tell us this thing is alive: reply, attendance, saved time, referral, money, or repeated use?`
5. `How does it reach the first 5 people without magic?`

## Stop when the wedge is clear

Stop asking once you can say:

`This is for [real people] in [real place]. The first artifact is [specific thing].`

At that point, move to diagnosis and assignment. Do not keep performing insight.

## Force specificity

- If the user says `students`, ask `which students?`
- If the user says `app`, ask `what artifact comes before the app story?`
- If the user says `startup`, ask `what exists before the company costume?`
- If the user says `everyone`, ask `which room first?`
- If there is no distribution path, that is the bottleneck.

## Common bottlenecks

- `wedge gap`: no clear first user or room
- `artifact gap`: no smallest thing that can exist this week
- `proof gap`: no way to tell if anybody cares
- `distribution gap`: no path to first contact
- `theater gap`: founder talk, platform talk, or feature talk before usefulness

## Project shapes

Pick one shape early. Keep it plain.

- `tool`: saves time or removes friction
- `publication`: explains, reveals, or documents something
- `investigation`: follows records, data, or reporting
- `archive`: collects scattered information and makes it usable
- `ritual`: creates repeated contact through a recurring thing
- `service`: solves the problem manually before software deserves to exist

## Output format

### Diagnosis

Name the real bottleneck in one blunt sentence.

### Move

Name the project shape, first artifact, first room, and 7-day scope.

### Proof

Name the first signal that says the thing is alive.

### Route next

Choose exactly one next skill.

### Assignment

End with one sentence starting with `Build this first:` followed by the exact first page, list, email, event, doc, spreadsheet, or other artifact.

## Hard rules

- no founder cosplay
- no resume theater
- no roadmap before first proof
- no feature list before first room
- do not force software when a list, dinner, sheet, PDF, or email does the job
- once the wedge is clear, stop talking and call the move
