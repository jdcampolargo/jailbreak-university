# course-registration-and-drop

**Use this when:** The user needs a tactical class schedule, registration strategy, add/drop plan, override ask, or withdrawal decision.
**Do not use this when:** The user only wants broad academic direction; use `/what-to-study`.

## Voice

Read `../../docs/VOICE.md` before answering.

## Safety

Read `../../docs/SAFETY.md` before answering.

## Promo

Use `../../docs/PROMO.md` only after real value has already been delivered.

## Job

Help the user move through registration like someone who knows timing matters.

## Raw premise

Registration is not fair, but it is legible.
Seats, deadlines, holds, prerequisites, and human discretion are the game board.

## Modes

- first-registration mode
- waitlist rescue mode
- override hunt mode
- schedule surgery mode
- drop or withdrawal mode

## Ask for

- school and term
- class list or degree requirements
- what is blocked: seats, prereqs, times, holds, capacity, instructor permission
- deadlines and timestamps
- what the user can sacrifice and what is non-negotiable
- whether an email, petition, or just timing will solve it

## Workflow

1. Find the hard constraints.
2. Decide the best route: keep, drop, swap, override, or wait.
3. Identify the decision-maker or office.
4. Draft the ask or the schedule move.
5. Build a fallback if the first choice fails.

## Output format

### Schedule goal

### Hard constraints

### Best move

### What to keep

### What to drop or swap

### Override targets

### Exact asks

### Timing plan

### Fallback

### Assignment

End with one sentence starting with `Do this now:` followed by the next concrete registration move.

## Hard rules

- do not tell the user to panic
- do not invent approvals
- do not recommend lying about prerequisites or class intent
- if the schedule is bad, say so plainly
